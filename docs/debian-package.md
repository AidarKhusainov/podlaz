# Debian package contract

This document defines the local Debian package layout, lifecycle behavior, validation targets, and inspection gates for TunWarden.

The package contract is intentionally limited to a locally installable `.deb` artifact. Public apt repository publication and repository signing are separate follow-up work. Tagged GitHub Release publication is owned by [Release workflow](./release.md) and consumes this package contract instead of redefining it.

## Supported validation targets

Initial package validation targets:

- Debian stable on `amd64`.
- Ubuntu LTS on `amd64`.

`arm64` is supported by the packaging script when the Go toolchain and build host can produce the target binary, but `arm64` must not block the first package acceptance gate.

Container validation is acceptable for package metadata, file layout, AppStream metadata, and basic install/remove checks. Full service lifecycle validation must be done on a VM or host where systemd is PID 1.

## Package toolchain

The package toolchain version contract is owned by `packaging/package-toolchain.env`.

CI, local package validation, and release automation must install the pinned `NFPM_VERSION` from that file instead of using `@latest`.

## Build contract

Build a local package with:

```bash
bash scripts/build-deb.sh
```

The default build emits:

```text
dist/tunwarden_0.0.0~dev-1_amd64.deb
```

Override the binary version, Debian package version, and architecture with:

```bash
TUNWARDEN_VERSION=0.1.3 TUNWARDEN_DEB_VERSION=0.1.3-1 TUNWARDEN_DEB_ARCH=amd64 bash scripts/build-deb.sh
```

`TUNWARDEN_VERSION` controls the application version reported by `tunwarden version`. `TUNWARDEN_DEB_VERSION` controls the full Debian package version rendered into nFPM's `version` field and validated with `dpkg-deb --field <package> Version`. When unset, `TUNWARDEN_DEB_VERSION` defaults to `${TUNWARDEN_VERSION}-1`.

The build requires:

- Go with the project-pinned toolchain from `go.mod`.
- The pinned `nfpm` version from `packaging/package-toolchain.env`.
- `gzip`.
- Debian package tools such as `dpkg-deb`.
- `appstreamcli` from the AppStream toolchain.
- `man-db`.

The build script prepares a temporary package root under `dist/package-root`. That directory is build output only and must not be committed. The script also renders a temporary `.nfpm.tunwarden.yaml` config in the repository root and removes it after package generation.

## Installed file layout

The package installs only packaged files under Debian/FHS-appropriate locations:

```text
/usr/bin/tunwarden
/usr/bin/tunwardend
/usr/lib/systemd/system/tunwardend.service
/usr/lib/sysusers.d/tunwarden.conf
/usr/share/bash-completion/completions/tunwarden
/usr/share/zsh/vendor-completions/_tunwarden
/usr/share/fish/vendor_completions.d/tunwarden.fish
/usr/share/metainfo/io.github.aidarkhusainov.tunwarden.metainfo.xml
/usr/share/man/man1/tunwarden.1.gz
/usr/share/man/man8/tunwardend.8.gz
/usr/share/doc/tunwarden/README.md
/usr/share/doc/tunwarden/LICENSE
/usr/share/doc/tunwarden/copyright
/usr/share/doc/tunwarden/changelog.Debian.gz
/usr/share/doc/tunwarden/docs/...
```

Shell completion files are generated by `tunwarden completion bash`, `tunwarden completion zsh`, and `tunwarden completion fish` at package build time.

The AppStream MetaInfo file is maintained at `packaging/linux/io.github.aidarkhusainov.tunwarden.metainfo.xml` and installed as `/usr/share/metainfo/io.github.aidarkhusainov.tunwarden.metainfo.xml`. It models TunWarden as a CLI-first `console-application`, uses component ID `io.github.aidarkhusainov.tunwarden`, declares metadata license `CC0-1.0`, declares project license `MIT`, provides the `tunwarden` binary, includes homepage, bugtracker, and VCS browser URLs, includes OARS content rating metadata, and includes release notes for the current package metadata version.

The MetaInfo release entries are deterministic, reviewable source files. When release/package automation updates packaged versions, the corresponding release entry must be maintained with concise notes and must not include private provider data, subscription data, generated configs, or raw IDs.

TunWarden does not currently ship or reference a software-center icon or `.desktop` launcher. A future icon or desktop launcher must be intentionally added, installed under the appropriate freedesktop paths, and validated in CI instead of being introduced only to improve visuals.

The package must not install packaged files under:

- `/usr/local`
- `/run/tunwarden`
- `/var/run/tunwarden`
- user home directories
- generated runtime configuration paths

Generated configs, transaction files, daemon sockets, PID/process state, and other daemon runtime outputs are not packaged files.

## Package metadata

The Debian package metadata is owned by `packaging/nfpm.yaml`.

Initial metadata contract:

- package name: `tunwarden`
- section: `net`
- priority: `optional`
- architecture: `amd64` by default, `arm64` optional
- maintainer: project maintainer metadata from the package manifest
- runtime dependencies: `libc6`, `systemd`, and `ca-certificates`
- AppStream component ID: `io.github.aidarkhusainov.tunwarden`
- AppStream component type: `console-application`

## Service install behavior

Package installation places packaged files, registers the unit metadata, and creates service identities when supported by the host package tools. Package installation must preserve the existing no-auto-start and no-auto-enable lifecycle contract and must not apply host runtime configuration.

## State ownership and lifecycle

| Category | Location | Owner | Package behavior |
| --- | --- | --- | --- |
| Packaged files | `/usr/bin`, `/usr/lib/systemd/system`, `/usr/lib/sysusers.d`, `/usr/share/bash-completion`, `/usr/share/zsh`, `/usr/share/fish`, `/usr/share/metainfo`, `/usr/share/man`, `/usr/share/doc/tunwarden` | Debian package manager | Installed, upgraded, and removed by `dpkg`/`apt`. |
| Daemon runtime state | `/run/tunwarden` | `tunwardend` through systemd `RuntimeDirectory=` | Volatile; created when the service starts; not shipped in the package. |
| Generated runtime configs | `/run/tunwarden/generated` | `tunwardend` | Runtime output only; not shipped in the package and not persistent source of truth. |
| Daemon persistent state | `/var/lib/tunwarden` | systemd `StateDirectory=` and daemon | Reserved for daemon-owned persistent state; not shipped as packaged files. |
| User intent/state | `$XDG_CONFIG_HOME/tunwarden`, `$XDG_STATE_HOME/tunwarden`, `$XDG_CACHE_HOME/tunwarden` | invoking user | Not owned, modified, or removed by package lifecycle scripts. |

## Inspection and validation gates

Package inspection:

```bash
dpkg-deb --info dist/tunwarden_0.0.0~dev-1_amd64.deb
dpkg-deb --contents dist/tunwarden_0.0.0~dev-1_amd64.deb
dpkg-deb --contents dist/tunwarden_0.0.0~dev-1_amd64.deb | grep -F './usr/share/metainfo/io.github.aidarkhusainov.tunwarden.metainfo.xml'
```

AppStream metadata validation:

```bash
appstreamcli validate --no-net packaging/linux/io.github.aidarkhusainov.tunwarden.metainfo.xml
appstreamcli validate --no-net /usr/share/metainfo/io.github.aidarkhusainov.tunwarden.metainfo.xml
```

Local install validation includes checking the installed binary, service metadata, shell completions, AppStream metadata, manual pages, and package remove behavior. Full service lifecycle checks require a VM or host where systemd is PID 1.

Use `lintian` where practical:

```bash
lintian dist/tunwarden_0.0.0~dev-1_amd64.deb
```

A package PR must document whether `lintian` was clean or list every relevant warning with justification.

## Container versus VM validation

Container validation can check:

- package metadata;
- package contents;
- AppStream MetaInfo validation;
- local install/remove mechanics;
- absence of `/usr/local`, `/run`, `/var/run`, user home, and generated runtime config packaged paths;
- binary, shell completion, AppStream metadata, and man page presence.

Container validation is not sufficient for:

- service status behavior when systemd is not PID 1;
- daemon startup under the packaged unit;
- journald integration;
- service runtime directory creation through systemd;
- any real networking lifecycle.

Those checks require a VM or systemd-capable host.
